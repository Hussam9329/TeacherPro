# تقرير باتش سلامة التسجيل والفصول ومزامنة الطلاب

## الخلاصة

المشكلات التسع المذكورة كانت منطقية ومؤثرة فعلاً على عمل المستخدم. بعضها كان يسبب فشل عملية واضحة، وبعضها كان يستطيع إنشاء حالة بيانات متعارضة أو فقدان سجل فرص عند فشل جزئي. تم إصلاحها ضمن باتش واحد يغطي قاعدة البيانات، مسارات الخادم، الواجهة، والتعامل مع الطلبات المتزامنة.

## تقييم المشكلات والإصلاح

| المشكلة | مستوى الضرر قبل الإصلاح | النتيجة بعد الباتش |
|---|---:|---|
| تسجيل طالبين في اللحظة نفسها | حرج: قد يتولد الكود نفسه ويفشل أحد التسجيلين | توليد الأكواد من PostgreSQL Sequence ذرية، مع حجز جماعي وإعادة محاولة تلقائية عند وجود كود قديم متعارض |
| تعديل إعدادات الدورة مع بقاء Snapshot الطلاب القديمة | مرتفع: اختلاف بين الإعدادات والطلاب بدون مسار تصحيح مباشر | معاينة أثر قبل الحفظ، وخيار حفظ الإعداد فقط أو مزامنة الطلاب المتأثرين، مع استثناء المؤرشفين |
| تغيير سقف فرص الفصل | حرج: السقف المعروض يتغير بينما الرصيد المخزن يبقى قديماً | معاينة عدد المتأثرين، ثم مزامنة اختيارية فورية وإعادة احتساب الحالة الأكاديمية من المصدر الحقيقي |
| إلغاء تفعيل الفصل الوحيد | حرج: تصفير واسع للطلاب | معاينة خادمية حديثة، إظهار عدد المتأثرين والأرشيف، ومنع التنفيذ إذا تغيّرت الحالة أو ظهر تعارض قبل التنفيذ |
| وجود أكثر من فصل نشط للدورة | حرج: بقية النظام يرفض العمل أو يختار حالة غير صالحة | قيد Unique جزئي في PostgreSQL يمنع التعارض حتى بين طلبات متزامنة، مع إصلاح آمن للتعارضات القديمة أثناء الترحيل |
| نقل ربط فصل نشط إلى دورة أخرى | حرج: يستطيع إنشاء تعارض في الدورة الجديدة وترك أثر قديم في المصدر | منع النقل المباشر للربط النشط؛ يجب إلغاء التفعيل ثم النقل ثم التفعيل عبر المسار الرسمي ذي المعاينة |
| مسار المزامنة يثق بقيمة الفرص المرسلة من العميل | حرج: عميل قديم أو طلب معدل يستطيع كتابة أرصدة خاطئة | الخادم يتجاهل قيمة العميل ويعيد قراءة الدورة، الفصل النشط، والسقف الحقيقي داخل نفس Transaction |
| إلغاء الأرشفة بلا قصد عند تعطيل روابط أخرى | مرتفع: استعادة سجل مؤرشف بدون إجراء صريح | التعطيل يغيّر `active` فقط ولا يكتب `archived:false` |
| نقل الطالب كطالب جديد خارج Transaction موحدة | حرج: فقدان سجل الفرص إذا فشل تحديث الطالب بعد الحذف | حذف سجل الفرص، تحديث الطالب، إعادة الاحتساب، وقراءة النتيجة كلها داخل Transaction واحدة |

## ضمانات التزامن الجديدة

- التسجيل الفردي والجماعي، نقل الطالب كطالب جديد، وتعديلات الدورة والفصل ومزامناتها الواسعة تعمل بعزل PostgreSQL `SERIALIZABLE`.
- عند حدوث Serialization conflict أو Deadlock يعيد الخادم العملية تلقائياً حتى 5 محاولات، ولا يعرض نجاحاً جزئياً.
- معاينة الواجهة ليست مصدراً للتنفيذ؛ الخادم يعيد قراءة الروابط والطلاب والقيم الحقيقية داخل Transaction وقت الحفظ.
- المؤرشفون لا تتم مزامنتهم تلقائياً، ويبقون كسجل تاريخي.

## ترحيل قاعدة البيانات

الترحيل الجديد:

`prisma/migrations/20260712190000_atomic_student_codes_and_active_chapter_guard/migration.sql`

ينفذ ما يلي:

1. إنشاء وتهيئة `Student_code_seq` فوق أعلى كود `BIO` موجود.
2. أرشفة روابط الدورة/الفصل القديمة المكررة بدلاً من حذفها.
3. إصلاح وجود أكثر من فصل نشط للدورة مع الإبقاء على أفضل ربط مطابق لأرصدة الطلاب الحالية، ثم تعطيل البقية دون حذفها.
4. إنشاء قيدين جزئيين يمنعان تكرار الربط غير المؤرشف ووجود فصلين نشطين للدورة.

## خطوات النشر المطلوبة

1. أخذ نسخة احتياطية من PostgreSQL قبل الترحيل.
2. تثبيت الاعتمادات بنفس نسخة المشروع:

```bash
npm ci
```

3. تطبيق الترحيلات **قبل فتح النسخة الجديدة للمستخدمين**:

```bash
npm run db:deploy
```

4. بناء وتشغيل المشروع:

```bash
npm run build
npm start
```

لا يُنصح بنشر كود الباتش قبل تنفيذ `npm run db:deploy`؛ القيود الجديدة والـSequence جزء أساسي من الضمانات.

## الاختبارات المنفذة

- نجح الأمر الكامل `npm run test:side-effects` بكل وحداته: المكالمات، الدورات، الفصول، الفرص، التسجيل الفردي والجماعي، سجل الطلاب، المفصولون، الامتحانات، الدرجات، الإجازات، التعهدات، الحسابات والصلاحيات، السجلات، المزامنة الذكية، واللغة وتجربة المستخدم.
- نجح ESLint لكل ملفات الباتش المعدلة.
- فحص TypeScript لم يظهر أخطاء في ملفات الباتش. بقيت أخطاء في ملفات أخرى غير معدلة عند استخدام تعريف Prisma مؤقت داخل بيئة الفحص.
- تعذر تنفيذ بناء إنتاج كامل داخل بيئة الفحص لأن `prisma generate` احتاج تنزيل محرك Prisma من `binaries.prisma.sh` والبيئة لا تملك اتصالاً خارجياً. هذا قيد للبيئة وليس خطأ ظهر في كود الباتش، لذلك يجب تشغيل `npm run build` في بيئة النشر المتصلة قبل تحويل الزيارات.

## ملاحظة الاسترجاع

يمكن الرجوع إلى كود النسخة السابقة عند الضرورة، لكن لا يُنصح بإلغاء أرشفة الروابط التي أصلحها الترحيل أو حذف القيود الجديدة يدوياً دون مراجعة نسخة قاعدة البيانات الاحتياطية. القيود نفسها متوافقة مع منطق النسخة الجديدة وتمنع عودة حالات التعارض.
